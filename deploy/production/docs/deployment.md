# syrnike13 Deployment

This directory contains the production Docker Compose based syrnike13 chat stack.

The deploy keeps runtime state on the server. GitHub Actions uploads repository files, but it does not overwrite:

- `data/`
- `.env`
- `.env.web`
- `secrets.env`
- `Syrnike.toml`
- `livekit.yml`
- `compose.override.yml`

Those files contain server-local state, generated config, or secrets.

## Required Server State

Use Ubuntu with Docker Engine and the Docker Compose plugin installed.

Open these ports:

- `80/tcp`
- `443/tcp`
- `7881/tcp`
- `50000-50100/udp`

If syrnike13 is behind an external reverse proxy, expose Caddy on `8880` through `compose.override.yml` generated by `generate_config.sh`, and proxy WebSocket paths `/ws` and `/livekit` correctly.

## GitHub Settings

Create the `production` environment in the `syrnike13/syrnike13-app` repository, then set:

Secrets:

- `DEPLOY_HOST`: server hostname or IP.
- `DEPLOY_USER`: SSH user.
- `DEPLOY_PATH`: absolute deploy directory, for example `/opt/syrnike13`.
- `SSH_PRIVATE_KEY`: private key allowed to SSH into the server.

Variables:

- `SYRNIKE_DOMAIN`: public domain. Use `syrnike13.ru` for production.
- `SYRNIKE_REVERSE_PROXY`: `false` for built-in Caddy on ports 80/443, `true` when another reverse proxy is in front.
- `SYRNIKE_VIDEO_ENABLED`: `true` to enable camera and screen sharing, `false` to disable.

## First Deploy

The first deploy generates server-local config by running:

```bash
./generate_config.sh "$SYRNIKE_DOMAIN"
```

The script answers the interactive reverse-proxy and video prompts from `SYRNIKE_REVERSE_PROXY` and `SYRNIKE_VIDEO_ENABLED`.

Back up `secrets.env` after the first successful deploy. Losing it can make existing uploaded files inaccessible.

## Updating Backend Services

Pushing to `main` runs `.github/workflows/docker.yaml`. When Docker image publishing succeeds, `.github/workflows/deploy-production.yml` uploads `deploy/production` and deploys backend-owned services:

- `api`
- `events`
- `autumn`
- `january`
- `gifbox`
- `crond`
- `pushd`
- `voice-ingress`

That workflow runs:

```bash
DEPLOY_SERVICES="api events autumn january gifbox crond pushd voice-ingress" ./scripts/deploy-on-server.sh
```

The server then runs:

```bash
docker compose pull api events autumn january gifbox crond pushd voice-ingress
docker compose up -d --no-deps api events autumn january gifbox crond pushd voice-ingress
docker compose ps api events autumn january gifbox crond pushd voice-ingress
```

## Updating The Full Stack

Run `.github/workflows/deploy-production.yml` manually with `full_stack=true` when production infrastructure files changed and the entire stack should be reconciled.

Without `DEPLOY_SERVICES`, the server runs:

```bash
docker compose pull
docker compose up -d --remove-orphans
docker compose ps
```

## Updating The Shared Edge

Production and nightly share the production Caddy container. Both deploy
workflows upload a candidate `Caddyfile` and run
`scripts/reload-caddy.sh`, which validates the candidate before copying it over
the bind-mounted file and applies it with `caddy reload`. The reload is locked
across both workflows and rolls the file back if Caddy rejects the new
configuration, so an edge update does not restart the container or disconnect
existing WebSocket and LiveKit sessions.

## Desktop Auto-Updates

Desktop installers and `electron-updater` manifests are served as static files from:

- URL: `https://<domain>/downloads/desktop/`
- Server path: `<DEPLOY_PATH>/data/desktop/`

Caddy serves that directory read-only from `data/desktop`. The directory is server-local state and is not overwritten by production deploy tar uploads.

Publishing flow:

1. Push a `VERSION` change on `main` to run `.github/workflows/desktop-release.yml`.
2. GitHub Actions builds Windows, macOS, and Linux packages.
3. The workflow uploads installers, `latest*.yml`, and `*.blockmap` files to `data/desktop/` on the production server.
4. Packaged desktop clients check `https://syrnike13.ru/downloads/desktop/` for updates.

Before the first desktop release upload, deploy production once so Caddy picks up the `/downloads/desktop/*` route and the `data/desktop` volume mount:

```bash
docker compose up -d --no-deps caddy
```

Health check:

```bash
curl -fsS "https://syrnike13.ru/downloads/desktop/latest.yml"
```

GitHub Releases remain available for manual downloads. Auto-update uses the production static feed.

## Stable Landing Download Aliases

The landing page links to stable URLs that always point at the latest desktop release:

- `https://<domain>/downloads/syrnike13.dmg`
- `https://<domain>/downloads/syrnike13-setup.exe`
- `https://<domain>/downloads/syrnike13.AppImage`

Caddy serves these from symlinks in `data/desktop/`. The desktop release workflow updates the symlinks from `latest*.yml` after each upload via `tooling/scripts/update-download-aliases.sh`.

Health check:

```bash
curl -fsSI "https://syrnike13.ru/downloads/syrnike13.dmg" | head -5
```

## Nightly Channel

The `develop` branch publishes and deploys the independent nightly environment.

- Public URL: `https://beta.syrnike13.ru`
- Server path: `/opt/syrnike13-nightly`
- Compose project: `syrnike13-nightly`
- Docker tags: `nightly` and `nightly-<sha>`
- Desktop delivery: GitHub prerelease `nightly`; no auto-update feed.

Production Caddy remains the edge proxy for both domains. It joins the shared
external Docker network `syrnike13-edge` and routes `beta.syrnike13.ru` to
nightly service aliases such as `syrnike13-nightly-web` and
`syrnike13-nightly-api`.

Nightly keeps its own server-local runtime files and data:

- `data/`
- `.env`
- `.env.web`
- `secrets.env`
- `Syrnike.toml`
- `livekit.yml`
- `compose.override.yml`

Nightly LiveKit uses separate host ports to avoid conflicts with production:

- `7882/tcp`
- `3479/udp`
- `30101-30200/udp`
- `50101-50200/udp`
